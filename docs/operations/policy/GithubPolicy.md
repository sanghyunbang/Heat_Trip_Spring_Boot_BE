
# 📌 GitHub 협업 전략 (GitHub Desktop 기반 3인 팀 전용 가이드)

---

## ✅ 전체 흐름 요약

```
1. 각 개발자가 맡은 기능을 GitHub Issue로 등록
2. 각 기능마다 개별 브랜치 생성 (예: feature/signup-form, feature/post-list, feature/comment-system)
3. 작업 진행 → WIP 커밋 반복 (Related to #이슈번호)
4. 매일 작업한 내용을 feat-wip 커밋 후 PR 생성 → main 브랜치에 병합
5. 다음날 main 브랜치 최신화 → 다시 내 브랜치에 병합하고 작업 이어감
6. 기능 완료 시 feat 커밋 + PR → Closes #이슈번호
7. 브랜치 삭제
```

---

## ✅ 브랜치 종류 및 명명 규칙

| 브랜치 종류      | 예시                              | 용도 설명                              |
|------------------|-------------------------------------|-----------------------------------------|
| 메인 브랜치      | `main`                             | 항상 배포 가능한 상태 유지              |
| 기능 브랜치      | `feature/signup-form`              | 개발자 A: 회원가입 폼 기능 개발         |
| 기능 브랜치      | `feature/post-list`                | 개발자 B: 게시글 목록 출력 기능 개발    |
| 기능 브랜치      | `feature/comment-system`           | 개발자 C: 댓글 작성/출력 기능 개발      |
| 버그 수정 브랜치 | `fix/navbar-overlap`               | 특정 버그 수정용                        |
| 디자인 브랜치    | `design/landing-page`              | 디자인 관련 작업                        |
| 실험/연구 브랜치 | `experiment/ai-recommendation`     | 실험적인 기능이나 임시 작업             |

## ✅ GitHub Desktop 기반 협업 순서

### 🔹 작업 시작 전: main 브랜치 최신화

1. GitHub Desktop 왼쪽 상단 `Current Branch`에서 `main` 선택  
2. 상단 툴바 → `Fetch origin` → `Pull origin` 클릭

---

### 🔹 내 브랜치로 돌아가 main 병합

1. `Current Branch` → 내 작업 브랜치 선택 (예: `feature/signup-form`)
2. 상단 메뉴 → `Branch > Merge into current branch`
3. `main` 선택 후 병합

---

### 🔹 작업 후 커밋 & Push

1. GitHub Desktop이 변경 사항 자동 감지
2. 커밋 메시지 입력 후 `Commit to [브랜치명]`
   ```
   [feat-wip]: 회원가입 UI 구현
   Related to #4
   ```
3. `Push origin` 클릭

---

### 🔹 PR 생성 및 병합

1. GitHub Desktop 오른쪽 상단 → `View on GitHub` 클릭
2. GitHub 웹에서 `Compare & Pull Request` 클릭
3. PR 설명 작성  
   - 기능 미완성 시: `Related to #이슈번호`  
   - 기능 완성 시: `Closes #이슈번호`
4. 팀원 리뷰 후 병합

---

### 🔹 다음날 작업 시

1. GitHub Desktop → `main` 선택 → `Pull origin`
2. 내 작업 브랜치로 돌아감 → `Branch > Merge into current branch` → `main` 선택
3. 최신 상태에서 작업 이어가기

---

## ✅ 커밋 메시지 규칙

```
[feat-wip]: 기능 일부 작업
Related to #이슈번호

[feat]: 기능 완성
Closes #이슈번호
```

---

## ✅ 협업 단계별 체크리스트

| 단계 | 체크할 내용 |
|------|-------------|
| 🧩 Issue 생성 | 기능 목적, 담당자, 체크리스트 포함 |
| 🔧 브랜치 생성 | `feature/기능명` 명확히 작성 |
| 🛠 커밋 반복 | WIP 커밋에 Related to #번호 사용 |
| 🔀 매일 병합 | PR을 생성하고 main에 병합하여 동기화 유지 |
| ✅ 기능 완성 | `feat` 커밋으로 마무리 + 최종 PR 생성 |
| 🔗 이슈 연결 | PR에 `Closes #번호` 작성 (기능 완성 시) |
| 🔁 병합 후 | 브랜치 삭제 및 main pull |

# ✅ 커밋 메시지 작성 가이드 (GitHub Desktop용)

---

## 🔹 커밋 타이틀 (Summary)

### 📌 형식
```
[타입]: 기능 요약 (짧고 명확하게)
```

### 📌 예시
```
[feat-wip]: 댓글 작성 UI 1차 구현
[feat]: 댓글 등록 기능 구현 완료
[fix]: 댓글 중복 등록 오류 수정
```

### 📌 타입 목록

| 타입       | 의미                             |
|------------|----------------------------------|
| feat       | 새로운 기능 추가                 |
| feat-wip   | 기능 개발 중간 상태 (미완성)     |
| fix        | 버그 수정                        |
| refactor   | 리팩토링 (기능 변화 없음)        |
| docs       | 문서 수정 (README 등)            |
| style      | 코드 포맷 변경, 세미콜론 등      |
| test       | 테스트 코드 추가                |
| chore      | 설정파일 등 기타 변경           |

---

## 🔹 커밋 설명 (Description)

> 템플릿 형식으로 작성하세요:

```
## 작업 내용
- 무엇을 개발했는지 핵심적으로 요약
- 어떤 이유로 이 변경이 필요한지

## 변경 항목
- [컴포넌트명] 무엇을 추가/수정/삭제
- [API] 어떤 API를 연결/변경

## 남은 작업
- (선택) 아직 미완성된 부분이나 TODO

## 관련 이슈
Related to #이슈번호  ← 미완성 기능
Closes #이슈번호     ← 완료 기능
```

## ✅ 실제 예시: 기능 중간 작업 (WIP)

### 📌 타이틀
```
[feat-wip]: 댓글 입력창 UI 1차 구현
```

### 📌 설명
```
## 작업 내용
- 댓글 입력창 레이아웃 구성 (input + 버튼)

## 변경 항목
- CommentForm 컴포넌트 생성
- Tailwind 적용

## 남은 작업
- API 연동 및 등록 로직 구현 예정

## 관련 이슈
Related to #7
```
