# 8. FAQ And Decisions

## Q1. 왜 native SQL로 바꿨나

기존 검색 어댑터는 JPA Criteria API를 사용했다.
하지만 이번 최적화의 핵심은 MySQL Fulltext 기능 사용이다.

즉 필요한 것은:

- `MATCH(...) AGAINST(...)`
- `BOOLEAN MODE`
- `WITH PARSER ngram`

이런 MySQL 전용 기능이다.

Criteria로도 억지로 표현할 수는 있지만,
이번 케이스에서는 오히려 native SQL이 더 단순하고 명확했다.

정리:

- Criteria 장점: DB 독립성, 자바 코드 조립
- native 장점: MySQL Fulltext 기능을 정확히 사용 가능

이번 작업은 검색 경로 하나만 바꾸는 것이므로
native SQL 사용에 따른 비용보다 얻는 이득이 더 컸다.

## Q2. 꼭 native로 바꿔야 했나

절대적으로는 아니다.
하지만 이번 요구사항에서는 사실상 가장 현실적인 선택이었다.

대안:

1. Criteria 유지
   - 가능은 하지만 코드가 더 복잡해질 수 있음

2. native SQL로 검색 경로만 전환
   - 이번 선택

3. Elasticsearch/OpenSearch 도입
   - 현재 단계에서는 과한 선택

## Q3. 다른 실행계획도 다 바꿔야 하나

아니다.
바뀐 경로만 새로운 SQL 기준으로 보면 된다.

구분:

- `q`가 있는 텍스트 검색
  - 이제 `MATCH ... AGAINST` 기준으로 본다

- `contentTypeId`, `cat3`, `emotionCategoryId` 같은 정형 필터
  - 여전히 정형 조건 인덱스 관점에서 분석 가능

다만 항상 주의할 점은,
"손으로 쓴 예시 SQL"이 아니라 "현재 코드가 실제로 만드는 SQL"을 기준으로 봐야 한다는 것이다.

## Q4. 다른 기능 깨질 가능성은 없나

영향 범위는 검색 API 중심이다.

직접 바뀐 것은:

- 검색용 컬럼 동기화
- 검색 SQL

직접 안 바뀐 것은:

- `scroll`
- 상세 조회
- 감정 API

즉 전체 서비스 위험도가 아주 큰 리팩터링은 아니다.

## Q5. `LIKE '%카페%'` 실행계획은 이제 의미가 없나

현재 백엔드가 더 이상 그 SQL을 사용하지 않는다면,
그 실행계획은 현재 코드의 설명 자료로는 의미가 없다.

다만 아래 용도로는 의미가 있다.

- 왜 기존 구조가 느렸는지 설명
- 최적화 전후 비교 자료
- 회고/문서화

즉 "과거 문제를 설명하는 근거"로는 유효하지만,
"현재 코드의 성능 자료"로는 쓰면 안 된다.

## Q6. 왜 `search_text`를 denormalize했나

매 요청마다 여러 컬럼, 다른 테이블 텍스트를 합쳐 검색하는 것보다
검색용 텍스트를 미리 준비해두는 편이 더 빠르고 단순하다.

검색은 정규화보다 검색 비용과 응답 속도가 우선되는 경우가 많다.
이번 작업이 그 전형적인 사례다.
